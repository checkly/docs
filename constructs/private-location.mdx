Use PrivateLocation to create private locations for running checks from your own infrastructure, VPCs, or segregated networks. This enables monitoring of internal services and applications not accessible from the public internet.

<Info>
Learn more about private locations in the [Private Locations documentation](/docs/introduction/platform/private-locations/).
</Info>

## `PrivateLocation` Examples

<CodeGroup>

```ts Basic Example
import { PrivateLocation } from 'checkly/constructs'

const myPrivateLocation = new PrivateLocation('private-location-1', {
  name: 'My Private Location',
  icon: 'server',
  slugName: 'my-private-location'
})
```

```ts Advanced Example
import { PrivateLocation, ApiCheck } from 'checkly/constructs'

const datacenterLocation = new PrivateLocation('datacenter-east-1', {
  name: 'East Coast Datacenter',
  icon: 'building',
  slugName: 'datacenter-east-1',
  proxyUrl: 'http://proxy.datacenter.local:8080'
})

// Use the private location in a check
new ApiCheck('internal-api-check', {
  name: 'Internal API Check',
  privateLocations: [datacenterLocation],
  request: {
    method: 'GET',
    url: 'https://internal-api.company.local/health'
  }
})
```

</CodeGroup>

## `PrivateLocation` Configuration

<Tabs>
  <Tab title="Basic Configuration">

Configure essential private location settings:

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `name` | `string` | ✅ | - | Friendly name for your private location |
| `slugName` | `string` | ✅ | - | Unique slug identifier (must be unique across your account) |
| `icon` | `string` | ❌ | `'server'` | Octicon name to distinguish the location in the UI |
| `proxyUrl` | `string` | ❌ | - | Proxy URL for outgoing HTTP calls from this location |

</Tab>
</Tabs>

      name: 'LDAP Service Health',

## Using Private Locations in Checks

Private locations can be used with all check types that support location configuration:

<Tabs>
  <Tab title="API Checks">
    ```ts
    new ApiCheck('private-api-check', {
      name: 'Private API Check',
      privateLocations: [privateLocation],
      // Can also combine with public locations
      locations: ['us-east-1'],
      request: {
        method: 'GET',
        url: 'https://internal.company.local/api/health'
      }
    })
    ```
  </Tab>
  
  <Tab title="Browser Checks">
    ```ts
    new BrowserCheck('private-browser-check', {
      name: 'Private Web App Check',
      privateLocations: [privateLocation],
      code: {
        entrypoint: path.join(__dirname, 'internal-app.spec.ts')
      }
    })
    ```
  </Tab>
  
  <Tab title="Multistep Checks">
    ```ts
    new MultiStepCheck('private-multistep-check', {
      name: 'Private Multistep Flow',
      privateLocations: [privateLocation],
      runtimeId: '2025.04',
      code: {
        entrypoint: path.join(__dirname, 'internal-workflow.spec.ts')
      }
    })
    ```
  </Tab>
  
  <Tab title="URL Monitors">
    ```ts
    new UrlMonitor('private-url-monitor', {
      name: 'Private URL Monitor',
      privateLocations: [privateLocation],
      request: {
        url: 'https://internal.company.local'
      }
    })
    ```
  </Tab>
  
  <Tab title="TCP Monitors">
    ```ts
    new TcpMonitor('private-tcp-monitor', {
      name: 'Private Database Connection',
      privateLocations: [privateLocation],
      request: {
        hostname: 'db.company.local',
        port: 5432
      }
    })
    ```
  </Tab>
</Tabs>

## Referencing Existing Private Locations

You can reference private locations created outside your CLI project:

<Tabs>
  <Tab title="By Slug Name">
    ```ts
    // Reference existing private location by slug
    new ApiCheck('existing-location-check', {
      name: 'Check Using Existing Location',
      privateLocations: ['existing-datacenter-slug'], // String reference
      request: {
        method: 'GET',
        url: 'https://internal.company.local/health'
      }
    })
    ```
  </Tab>
  
  <Tab title="Mixed References">
    ```ts
    // Combine new and existing private locations
    const newLocation = new PrivateLocation('new-location', {
      name: 'New Private Location',
      slugName: 'new-location'
    })

    new ApiCheck('mixed-locations-check', {
      name: 'Check with Mixed Locations',
      privateLocations: [
        newLocation,                    // New location object
        'existing-location-slug'        // Existing location by slug
      ],
      request: {
        method: 'GET',
        url: 'https://internal.company.local/health'
      }
    })
    ```
  </Tab>
</Tabs>

## Agent Deployment

After creating a private location, you'll need to deploy the actual agent:

<Info>
**Agent Installation**: After deploying your private location construct, you'll receive installation instructions and authentication keys through the Checkly UI or CLI output.
</Info>

<Warning>
**Unique Slug Names**: The `slugName` must be unique across your entire Checkly account. Choose descriptive, stable names that won't conflict with future locations.
</Warning>

## Best Practices

<Tabs>
  <Tab title="Naming Convention">
    ```ts
    // Good: Descriptive and consistent
    const eastDatacenter = new PrivateLocation('datacenter-east-1', {
      name: 'East Coast Datacenter',
      slugName: 'datacenter-east-1'
    })

    const devK8s = new PrivateLocation('k8s-dev-cluster', {
      name: 'Development Kubernetes',
      slugName: 'k8s-dev-cluster'
    })

    // Avoid: Generic or changing names
    const location1 = new PrivateLocation('loc-1', {
      name: 'Location 1',
      slugName: 'location-1'
    })
    ```
  </Tab>
  
  <Tab title="Logical Organization">
    ```ts
    // Organize by function/purpose
    const apiGateway = new PrivateLocation('api-gateway-location', {
      name: 'API Gateway Zone',
      icon: 'workflow',
      slugName: 'api-gateway-zone'
    })

    const databaseCluster = new PrivateLocation('database-cluster-location', {
      name: 'Database Cluster Zone',
      icon: 'database',
      slugName: 'database-cluster-zone'
    })
    ```
  </Tab>
